BEM (Block Element Modifier)

BEM provides a structured approach to CSS with clear naming conventions that prevent specificity issues.

BEM Structure

/* Block */
.card { ... }

/* Elements */

.card__title { ... }
.card__body { ... }

/* Modifiers */

.card--compact { ... }

OOCSS (Object-Oriented CSS)

OOCSS separates structure from skin and container from content for maximum reusability.

Standard Media Object

The media object is a classic OOCSS pattern. It separates the layout structure from the visual appearance.

Large Media Object

Same structure, different skin. The --large modifier changes spacing without affecting the core layout.

OOCSS Principles

  • Separate Structure from Skin: .media (structure) + .media--large (skin)
  • Separate Container from Content: Content adapts to any container
  • Minimize Selectors: Use classes instead of descendant selectors
  • Extend Objects: Build complex interfaces from simple objects

SMACSS (Scalable and Modular Architecture)

SMACSS categorizes CSS rules into Base, Layout, Module, State, and Theme categories.

Modules & States

SMACSS Categories

  • Base: Default styles (h1, p, body)
  • Layout: Major sections (.l-container, .l-sidebar)
  • Module: Reusable components (.button, .card)
  • State: State changes (.is-active, .is-hidden)
  • Theme: Look and feel (.theme-dark)

Atomic Design Hierarchy

  • Atoms: Basic HTML elements (.atom-input, .atom-label)
  • Molecules: Groups of atoms (.molecule-form-field)
  • Organisms: Groups of molecules (.organism-navbar)
  • Templates: Page-level components
  • Pages: Specific instances of templates

Architecture Comparison

Each methodology has strengths for different project types and team sizes.

When to Use BEM

  • Large teams needing clear conventions
  • Long-term maintenance projects
  • Complex component hierarchies
  • When avoiding specificity issues is crucial

When to Use Utility-First

  • Rapid prototyping and development
  • Consistent design systems
  • Small to medium teams
  • When bundle size optimization matters

When to Use OOCSS

  • High reusability requirements
  • Consistent UI patterns
  • Performance-critical applications
  • When CSS size matters

When to Use Atomic Design

  • Design system development
  • Component library creation
  • Cross-platform consistency
  • Documentation-heavy projects
